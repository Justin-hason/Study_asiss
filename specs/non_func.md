# Spec 3.8（修订版）：非功能需求实现策略规格  
（适配混合部署模式，覆盖裸机组件监控、备份容灾、性能与扩展性调整）

## 1. 目标
定义智能学习助手系统在混合部署环境（关键数据服务裸机/VM + 无状态服务容器化）下的非功能需求实现策略，确保系统在性能、召回率、防幻觉、安全性、可用性、可扩展性、可观测性及合规隐私等方面达到生产级质量标准，并提供针对裸机组件的备份、容灾与监控方案。

## 2. 非功能需求清单（不变）
- **性能**：检索 P95 <500ms，问答 SSE 首块 <2s，API 列表 <200ms，文档处理 <10s/MB
- **召回率**：RAG Top-5 召回率 ≥90%
- **防幻觉**：机制级杜绝捏造，无上下文 100% 拒绝
- **安全性**：多租户隔离、传输/存储加密、身份认证、防注入
- **可用性**：核心服务 99.9%，消息不丢失，故障自动转移
- **可扩展性**：万级到百万级文档平滑扩展
- **可观测性**：全链路监控、集中日志、分布式追踪、告警
- **合规与隐私**：数据隔离、生命周期管理、备份与数据删除

## 3. 设计细节与实现策略（修订后）

### 3.1 性能保障（混合部署优化）
- **检索链路**：裸机 Milvus/ES 提供极低延迟（<50ms），search-service 容器化并行调用，重排序批量推理。P95 预期 <300ms。
- **缓存策略**：
  - Redis（裸机）缓存高频问题答案与向量检索结果，TTL 根据文档更新频率自适应。
  - 权限缓存、会话缓存均在裸机 Redis，容器化服务直连，延迟 <1ms。
- **GPU 资源**：Embedding 和重排序服务运行在 GPU 节点，通过 K8s 资源配额确保不受其他容器干扰。
- **降级**：重排序超时则直接使用融合排序结果；外部大模型不可用时切换备用模型或返回友好提示。

### 3.2 召回率保障（不变）
- 混合检索（向量+BM25）+ Cross-Encoder 重排序
- 查询扩展（HyDE）与分块重叠
- 基于用户反馈的阈值与权重持续调优
- 定期离线评估，Recall@5 ≥90%

### 3.3 防幻觉机制强化（不变）
- 前置拦截（空上下文直接拒绝）+ 强制 Prompt 约束 + 后置语义审核
- 用户反馈闭环持续优化拦截阈值

### 3.4 安全与隐私（混合部署强化）
- **租户隔离**：所有查询强制 `tenant_id`，Milvus 分区、ES 过滤、数据库行级安全。
- **传输加密**：外部 HTTPS，内部服务间敏感通信使用 mTLS（Service Mesh 或手动证书）。
- **存储加密**：对象存储 AES-256，PG 透明数据加密（TDE），备份文件加密。
- **密钥管理**：K8s Secret 管理容器密钥，裸机服务使用 HashiCorp Vault 或本地加密配置。
- **审计**：敏感操作（权限变更、文档删除、管理员操作）写入不可变审计日志（ELK 或专门审计表）。

### 3.5 可用性与可靠性（混合部署方案）
- **核心组件高可用**：
  - Milvus/ES/Kafka/PG/Redis 均为集群部署，具备自动故障转移，无需依赖 K8s 编排。
  - 容器化服务多副本 + K8s 自动重启。
- **故障转移与降级**：
  - 数据库主从切换通过 Patroni 或手工脚本，网关自动重连。
  - 消息队列（Kafka）持久化 + 消费者组 rebalance 保证任务不丢。
- **备份与容灾（新增细节）**：
  - **数据库备份**：PostgreSQL 每日全量备份 + WAL 归档，异地存储（MinIO 其他区域或 S3 Glacier）。备份保留 30 天。
  - **向量库备份**：Milvus 元数据备份 + 底层存储快照（如 Ceph 快照）。
  - **ES 备份**：Snapshot 到共享文件系统或 S3，每日增量。
  - **灾难恢复**：RTO ≤ 4 小时，RPO ≤ 1 小时。定期恢复演练（每季度）。
- **容器化服务可靠性**：liveness/readiness 探针，资源限额防止雪崩。

### 3.6 可扩展性（混合模式路径）
- **数据层垂直/水平扩展**：
  - Milvus 添加数据节点，ES 添加节点，PG 读写分离与分片（按 tenant_id 分区）。
  - Kafka 增加 broker 与分区。
- **应用层水平扩展**：所有容器化服务通过 K8s HPA 或手动扩缩。
- **百万文档验证**：定期压测，验证线性扩展能力。

### 3.7 可观测性与监控（覆盖裸机）
- **指标监控**：
  - 裸机组件：通过 JMX Exporter（Kafka）、Prometheus Exporters（PG、Redis）、Milvus 内置指标、ES 的 Metricbeat 采集。
  - 容器服务：Prometheus 自动发现抓取 `/metrics`。
  - 统一 Grafana 仪表板，展示全栈指标。
- **日志聚合**：
  - 容器日志由 Filebeat 采集；裸机服务通过 Filebeat 读取本地日志文件。
  - 全部发送至 ELK/EFK 集群，索引按租户或服务分隔。
- **分布式追踪**：
  - API 网关注入 trace context，Jaeger 追踪请求在 K8s 服务间的流转；裸机服务间的调用需在客户端库中传递 trace header。
- **告警规则**：
  - P95 检索延迟 > 500ms，API 错误率 >1%，Kafka 消费者 lag > 阈值，节点宕机，磁盘利用率 >85% 等。
  - 告警推送至企业微信/钉钉/PagerDuty。

### 3.8 数据保护与合规
- 数据最小化，用户行为可匿名化。
- 数据生命周期：事件日志 TTL 180 天自动清理；删除文档时异步清理向量库和 ES 索引；账户注销后 30 天内彻底删除数据。
- 备份加密且异地存储，满足数据恢复要求。

## 4. 验证与交付标准（不变）
- 性能验收通过压力测试。
- 召回率评测 ≥90%。
- 防幻觉 100% 拒绝无匹配问题。
- 安全渗透测试无高危漏洞，租户隔离有效。
- 扩展验证：从 1 万到 50 万文档延迟增长 ≤20%。
- 可靠性：30 天核心服务 ≥99.9%。
- 灾难恢复演练成功（RTO/RPO 达标）。

## 5. 混合部署注意事项总结
- 裸机组件是性能与可靠性的基石，需配备专项运维手册和备份策略。
- 容器化服务与裸机组件间网络延迟需验证，推荐同机房/低延迟链路。
- 统一监控与日志体系是混合部署成功的关键，确保无监控死角。
- 所有部署配置标准化，使用 Infra as Code（Ansible/Terraform）管理裸机，Helm/Kustomize 管理 K8s。
