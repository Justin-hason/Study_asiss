# 代码审查检查清单

## 代码审查流程

```
PR 创建 → 自动检查（CI）→ 人工审查 → 反馈修正 → 批准合并
```

---

## 一、代码质量检查

### 1.1 可读性
- [ ] 变量、函数、类命名是否清晰有意义
- [ ] 代码结构是否清晰，逻辑是否易于理解
- [ ] 复杂逻辑是否有适当注释说明

### 1.2 简洁性
- [ ] 是否有冗余代码或重复逻辑
- [ ] 是否可以简化复杂的表达式
- [ ] 是否移除了调试代码和无用的日志

### 1.3 一致性
- [ ] 代码风格是否与项目一致
- [ ] 命名规范是否统一
- [ ] 错误处理模式是否一致

---

## 二、功能正确性检查

### 2.1 需求匹配
- [ ] 代码是否实现了需求描述的功能
- [ ] 是否有遗漏的功能点
- [ ] 是否存在未使用的代码

### 2.2 边界情况
- [ ] 是否处理了空值/空指针
- [ ] 是否处理了异常输入
- [ ] 是否处理了并发场景

### 2.3 错误处理
- [ ] 是否正确捕获和处理异常
- [ ] 是否有适当的错误日志
- [ ] 用户是否能理解错误信息

---

## 三、安全性检查

### 3.1 输入验证
- [ ] 是否验证了所有外部输入
- [ ] 是否防止了注入攻击（SQL/XSS/命令注入）
- [ ] 文件路径是否进行了规范化处理

### 3.2 数据保护
- [ ] 敏感数据是否加密存储
- [ ] API 密钥是否通过配置管理
- [ ] 是否避免将敏感信息记录到日志

### 3.3 权限控制
- [ ] 是否正确检查用户权限
- [ ] 是否存在越权访问风险
- [ ] 多租户隔离是否正确

---

## 四、性能检查

### 4.1 效率
- [ ] 是否有不必要的数据库查询
- [ ] 是否使用了合适的数据结构
- [ ] 是否存在潜在的性能瓶颈

### 4.2 资源管理
- [ ] 是否正确释放资源（文件句柄、数据库连接等）
- [ ] 是否有内存泄漏风险
- [ ] 缓存策略是否合理

---

## 五、测试检查

### 5.1 测试覆盖
- [ ] 是否有对应的单元测试
- [ ] 测试是否覆盖主要场景
- [ ] 是否有集成测试

### 5.2 测试质量
- [ ] 测试用例是否清晰易懂
- [ ] 是否包含边界条件测试
- [ ] 测试是否可重复执行

---

## 六、架构与设计检查

### 6.1 设计原则
- [ ] 是否遵循单一职责原则
- [ ] 是否符合开闭原则
- [ ] 模块间耦合是否合理

### 6.2 架构一致性
- [ ] 是否符合项目架构规范
- [ ] 是否使用了项目标准的设计模式
- [ ] API 设计是否符合 RESTful 原则

---

## 七、文档检查

### 7.1 代码文档
- [ ] 公共 API 是否有文档注释
- [ ] 复杂函数是否有使用说明
- [ ] 参数和返回值是否有说明

### 7.2 变更文档
- [ ] 是否更新了相关文档
- [ ] 是否添加了变更日志
- [ ] API 变更是否向后兼容

---

## 审查结论

| 状态 | 说明 |
|------|------|
| 🟢 批准 | 代码符合标准，可合并 |
| 🟡 有条件批准 | 需要修改特定问题后可合并 |
| 🔴 拒绝 | 需要重大修改 |
