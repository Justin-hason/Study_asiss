# 代码质量指南

## 1. 概述

本指南定义了智能学习助手系统的代码质量标准和最佳实践，确保代码的可维护性、可读性和可靠性。

## 2. 代码风格规范

### 2.1 Go 代码风格
- 遵循 Go 官方风格指南
- 使用 `gofmt` 进行自动格式化
- 使用 `golint` 或 `golangci-lint` 进行静态分析

### 2.2 Python 代码风格
- 遵循 PEP 8 规范
- 使用 `black` 进行自动格式化
- 使用 `flake8` 或 `pylint` 进行静态分析

### 2.3 通用规则
- 变量命名使用驼峰式或下划线式，保持一致性
- 函数/方法命名清晰，使用动词开头
- 类命名使用 PascalCase
- 常量使用全大写，下划线分隔

## 3. 代码审查清单

### 3.1 功能性检查
- [ ] 代码是否正确实现了需求
- [ ] 是否有遗漏的边界情况
- [ ] 是否处理了错误和异常

### 3.2 安全性检查
- [ ] 是否存在 SQL 注入风险
- [ ] 是否存在 XSS 漏洞
- [ ] 敏感数据是否加密处理
- [ ] 是否正确进行权限校验

### 3.3 性能检查
- [ ] 是否有不必要的数据库查询
- [ ] 是否使用了合适的索引
- [ ] 是否有内存泄漏风险

### 3.4 可维护性检查
- [ ] 代码是否有适当的注释
- [ ] 函数/方法是否过于复杂（圈复杂度 < 15）
- [ ] 是否遵循单一职责原则

## 4. 测试规范

### 4.1 测试覆盖率
- 单元测试覆盖率 ≥ 80%
- 关键业务逻辑必须有测试覆盖
- 使用 `go test` 或 `pytest` 进行测试

### 4.2 测试类型
- **单元测试**：测试单个函数/方法
- **集成测试**：测试多个组件协作
- **端到端测试**：测试完整业务流程

### 4.3 测试命名规范
- 测试函数命名：`Test[Feature]_[Scenario]`
- 示例：`TestSearchService_VectorSearch`

## 5. 持续集成检查

### 5.1 CI 流水线检查项
- [ ] 代码格式化检查
- [ ] 静态分析检查
- [ ] 测试覆盖率检查
- [ ] 安全扫描
- [ ] 依赖安全检查

### 5.2 门禁条件
- 所有检查必须通过才能合并代码
- 代码审查至少需要 1 位审核人批准
- 测试覆盖率不得低于基线

## 6. 代码质量度量指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 代码覆盖率 | ≥ 80% | 单元测试覆盖 |
| 圈复杂度 | < 15 | 函数复杂度 |
| 代码重复率 | < 5% | 重复代码检测 |
| 静态分析问题 | 0 | 严重级别问题 |
| 安全漏洞 | 0 | 高危漏洞 |

## 7. 工具推荐

### 7.1 静态分析工具
- **Go**: `golangci-lint`, `staticcheck`
- **Python**: `flake8`, `pylint`, `bandit`

### 7.2 代码格式化工具
- **Go**: `gofmt`
- **Python**: `black`, `isort`

### 7.3 测试工具
- **Go**: `go test`, `testify`
- **Python**: `pytest`, `coverage.py`

### 7.4 安全扫描工具
- **SAST**: SonarQube, Semgrep
- **依赖扫描**: Dependabot, Snyk

## 8. 最佳实践

1. **小步提交**：每次提交只做一个逻辑改变
2. **自文档化代码**：代码应该清晰易懂，注释解释"为什么"而非"是什么"
3. **防御性编程**：对输入进行验证，处理所有可能的错误情况
4. **代码复用**：提取公共逻辑到可复用的函数/模块
5. **定期重构**：技术债务需要定期清理
